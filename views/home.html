Access

An accessibility endeavor focusing on the four outlined areas of disability affecting web use: visual, hearing, motor, and cognitive.

Access is a tool to educate oneself about accessible HTML application & technique. By laying out our markup with accessibility as a guide, we positively affect other areas of development and broaden our view of the web as a whole.

This project's purpose is to provide code examples that help in bringing web sites closer to WCAG 2 compliance and are screen reader friendly. Currently, Access has only been tested with the NVDA screen reader (which is free & open-source).

Accessibility Overview

Visual

The visual aspect of accessibility is all about screen reader compatibility, which is controlled via keyboard functionality. Ensuring proper semantic markup is key in aiding visually-impaired users.

Hearing

The hearing aspect of accessibility is focused on audio-enabled media alternatives such as transcripts for videos, and ensuring the UI does not rely on audio-based cues or instructions.

Motor

The motor aspect of accessibility is greatly aided by ease-of-access, generally via keyboard functionality that is manipulated by assistive technologies.

Cognitive

The cognitive aspect of accessibility is tied to the organization and understandability of pages. Media that assists in content comprehension is of great help to individuals facing cognitive challenges.

Test Case: Navigation Bar

Arguably, the most important area of a web site is the navigation. The navigation allows for quick access to all site pages and is an invaluable asset to all users.

Universal Navigation (top of page)

    Expected Elements:
  • Skip to content link (hidden, tabbable)
  • Logo Link (tabbable)

Desktop Navigation (top of page)

    Expected Elements:
  • Standard Links (tabbable)
  • Dropdown Menus via Link Hover (hidden, expandable, tabbable)

Mobile Navigation

    Expected Elements:
  • Menu Button Trigger (triggers hidden menu, accessible, tabbable)
  • Mobile Menu (hidden, expandable, tabbable)

Test Case: eCommerce

The initial motivation for Access was based upon making essential eCommerce pages accessible. In this case, we're testing catalog pages and individual product pages.

Product Catalog

    Expected Elements:
  • Filter menu (hidden elements, sub-navigation)
  • Images (alt descriptions)
  • Favorites feature (icon with description)
  • Pagination (screen-reader friendly)

Single Product

    Expected Elements:
  • Main & alternate images (alt descriptions)
  • Social Sharing pop-out menu (hidden elements)
  • Information-filled Accordions (hidden elements)
  • CTA Buttons: Add to Cart, Add to Wishlist (explicit dependency on information accuracy via screenreader)
